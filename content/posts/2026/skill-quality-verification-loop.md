## 從「通過測試」到「真實可用」：AI Skill 品質驗證的實踐之路

在構建面向領域專家（律師、醫生、工程師）的 AI Skill 和 Plugin 時，一個常見的陷阱是：單元測試全部通過，上線後卻在真實場景中失效。這背後有三個隱藏的原因。

### 為什麼測試不夠

首先，**Prompt 的覆蓋缺口只在真實使用時才會浮現**。一個看似完整的 Skill 可能在特定領域的邊界案例上徹底失手——比如 NDA 審查工具漏掉了「吹哨者保護條款」，單元測試卻永遠驗證不出來。

其次，**邊界案例需要領域知識才能發現**。光靠工程師的想像力寫不出真實律師遇到的場景。你寫的測試案例可能是：「檢查文本是否包含『法定強制揭露』」——這確實驗證了關鍵詞的存在，但完全驗證不了系統是否能在實際合約中正確識別這類條款。

第三，**「11 點檢查清單通過」不等於「領域專家覺得有用」**。這是最扎心的一點。一份通過所有形式驗證的 Skill，專業人士用一次就會發現它遺漏了某個關鍵場景。

### 解決方案：驗證迴圈

最有效的方法是引入一個結構化的**驗證迴圈**，需要 3 輪或更多輪，目標是達成「某一輪不再發現新問題」。

#### 每一輪的流程

**第 N 輪：**

1. **進行真實案例研究** — 查閱業界新聞、法庭判例、產業報告，找出該領域的實際場景
2. **模擬完整互動流程** — 用域專家的語言描述需求 → AI 路由到相應 Skill → Skill 處理 → 生成回應
3. **逐項檢查 Skill 能力** — 不只是「Skill 有沒有觸發」，而是「Skill 是否正確識別了這個場景中的風險」
4. **以表格記錄發現** — 場景、能力項、是否檢測到、嚴重程度
5. **用 TDD 修復缺口** — 先寫失敗的測試，再實現修復
6. **迭代直到零新問題** — 重複上述步驟，直到某一輪不再發現新問題

**關鍵原則**：每一輪使用來自**不同子領域**的**全新場景**，而不是反覆測試同一類案例。

#### 實踐案例：NDA 審查工具的 3 輪驗證

以一個 NDA（保密協議）審查 Skill 為例，看看如何從有缺陷逐步迭代到可信賴：

| 輪次    | 涵蓋的場景                                                                                           | 發現新問題數 | 修復版本                                                                      |
| ------- | ---------------------------------------------------------------------------------------------------- | ------------ | ----------------------------------------------------------------------------- | ---- |
| 第 1 輪 | 科技新創員工、創投人士、併購談判、跨境交易、生科公司、製造業、娛樂業、HR、房地產、SaaS (10 個)       | 5 項         | 強化必要揭露檢測、IP 轉讓識別、隱性競業禁止、NDA 類型風險分類、罰款合理性評估 | v2.2 |
| 第 2 輪 | 科技員工完整流程、新創完整流程、YouTuber 完整流程、保險業代理完整流程、台積電洩密案例完整流程 (5 個) | 2 項         | 吹哨者保護例外識別、責任免除條款檢測                                          | v2.3 |
| 第 3 輪 | 政府採購、醫美、加盟合約、學術機構、離婚 NDA (5 個)                                                  | 0 項         | ✅ 驗證完成                                                                   |

這個表格展示的不只是進度，而是一個關鍵洞察：**直到第 3 輪，才發現之前完全沒想到的領域（政府採購、醫美、加盟）**。如果只做第 1 輪就上線，會漏掉這些完整的用戶群。

### 模擬流程範本

在進行驗證時，可以按照這個結構化範本進行：

```
**用戶（領域專家）**: [自然語言描述需求，就像真人在對話]

**AI 內部路由**:
  1. 偵測關鍵詞 → 路由到相應 Skill
  2. 預檢查（例如假 NDA 識別）
  3. 執行主要分析
  4. 生成回應

**能力檢查表**:
| 檢查項目 | 期望行為 | Skill 能否檢測 | 嚴重程度 |

**AI 對用戶的回應**: [完整回覆，包含分析結果和建議]

**發現的問題**: [描述 SKILL.md 的具體缺失，或標記 ✅ 已通過]
```

這個範本的好處是**強制你逐項驗證**，而不是籠統地「試試看效果如何」。表格會暴露出任何遺漏。

### 何時採用這套方法

這套驗證迴圈適用於幾個具體場景：

- **新建或大幅修改一個 AI Skill、Plugin 或 Prompt** 時，將其作為質量關卡
- **該 Skill 面向領域專家**（律師、醫生、財務顧問），他們的邊界案例知識超過一般用戶
- **單元測試全部通過，但你仍不確定真實覆蓋率** 時
- **作為上線前的「畢業要求」** — 0 新問題發現才能標記為生產就緒

換句話說，**如果一個 Skill 只服務非專家用戶（如通用寫作助手），這套方法可能過度工程化**。但如果你的 Skill 在法律、醫療、財務等需要專業判斷的領域運作，這套驗證迴圈會是上線前最後一道防線，也往往是最高效的防線。
