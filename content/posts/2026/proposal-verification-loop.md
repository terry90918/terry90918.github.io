在進行大型架構重構時，我們常常會遇到一個看似簡單卻容易失控的問題：一份複雜的提案文件看起來面面俱到，但往往在實作到一半時，才發現邏輯衝突、細節遺漏，或者目標早已偏離了最初的想法。更糟糕的是，每次修正都可能引入新的不一致，形成一個無止境的修補循環。

如果你的提案涵蓋超過 10 個 tasks，或者牽連 10+ 個檔案的修改，一次性的檢查根本找不完所有問題。那麼，有沒有什麼系統化的方法能夠有效地驗證提案的品質呢？

## 五輪迭代驗證法

經過多次在 OpenSpec 和大型重構中的實踐，我發現「五輪迭代驗證」這個方法特別有效。每一輪都聚焦於不同的面向，逐步逼近問題的各個角落。

### 第一輪：交叉一致性檢查

首先，你需要確保所有的文件互相印證。打開 proposal、design 和 specs 這三個文件，逐一檢查：

- **proposal ↔ design**：描述是否一致？是否有地方說法不同，暗示著邏輯混亂？
- **design ↔ specs**：設計中的每一個決策，是否都在 specs 中有對應的測試場景（GIVEN/WHEN/THEN）？
- **specs ↔ tasks**：每一個需求、每一個 spec，是否都對應到一個 task？反過來，每個 task 是否都能追蹤回某個需求？

這輪檢查的目的是找出「說了卻沒做」或「做了卻沒說」的部分。

### 第二輪：程式碼可行性驗證

理論上完美的提案，往往在面對實際程式碼時崩潰。所以第二輪要動手：拉開你的編輯器，用 `grep` 在實際程式碼中搜索。

檢查清單包括：

- 被呼叫的函數真的存在嗎？簽名是不是提案中描述的那樣？
- import 關係是否合理？會不會造成循環依賴？
- constructor 的簽名是否與提案中的使用方式匹配？

同時，要找出提案中沒有提到、但會被影響的檔案。比如 auto-repair.ts 的初始化邏輯、index.ts 的 re-export 清單。這些「沒列出來」的檔案往往藏著最多的坑。

### 第三輪：目標對齊檢查

回到最初。是什麼觸發了這個提案？「支援 OpenAI embedding」？「改進錯誤恢復機制」？「統一資料庫連線層」？

現在，列出所有的目標，然後逐一確認提案是否完整涵蓋。如果有遺漏，不要留下「未來工作」這樣的標記 — 如果這是目標的一部分，就在現在納入提案。

在大型重構中，「留給下次迭代」往往意味著它會被永遠遺忘，或者在後續工作中造成意想不到的複雜度。

### 第四輪：邊界情況掃描

想像系統在真實環境中運作時會發生什麼：

- 當多個 embedding model 並行請求時，誰先誰後？有沒有 race condition？
- 當某個步驟 crash 後，系統該如何恢復？是重新開始還是繼續從中斷點？
- 如果 PostgreSQL 暫時不可用，有沒有 graceful degradation 的策略？或者會直接拋出例外？

這些邊界情況往往在設計階段被忽略，卻在生產環境中造成最多的災難。

### 第五輪：命名與位置確認

最後一輪看起來最瑣碎，但也最容易出錯。確認以下細節：

- 新資料表該放在哪個資料庫？是產品資料庫還是共用資料庫？
- Migration 檔案放在哪個目錄？現在用的是 `migrations/` 還是 `migrations-shared/`？
- 新建的檔案如何命名？用時間戳記還是序號？有沒有既有的慣例？

這些看似小事的決定，會影響未來六個月的維護成本。不在現在確認清楚，就等著埋坑。

## 實踐要點

在執行這五輪驗證時，有三個關鍵的紀律值得強調：

**第一，每次修正後重新讀取全部 artifacts。** 不要假設「其他部分沒問題」。你在 proposal 中修改了一句話，可能會影響 design 中的整個決策樹。用工具輔助（Ctrl+Shift+F 或 grep），但親眼讀一遍。

**第二，用工具驗證，不要相信記憶。** 你可能記得「embed cache 的查詢是在 db.ts 裡」，但實際可能在 cache-layer.ts 或 embedding-service.ts。開啟檔案，搜索，確認。記憶是騙人的。

**第三，「不要有規劃未來在做的事」。** 如果增加 Anthropic embedding 支援是這個提案的目標之一，就現在納入 design。不要說「v2 會支援」。在大型重構中，v2 經常遙遙無期，或者因為基礎設施的變化而需要重新設計。

最後，每當你在修改設計時，問自己一個問題：「這個設計是在解決提案提出的問題，還是在因為完美主義而增加複雜度？」很多時候，最優雅的設計反而來自對「夠好」的理解。

## 何時適用

這套五輪驗證法在以下情況特別有價值：

- 你的提案已經産生了 proposal、design、specs 這三個以上的 artifacts
- 大型重構涉及 10+ 個檔案的修改
- 多人協作或多個 agent 共同完善設計文件
- 提案已經歷了 2 次以上的重大修改

如果你的變更只是簡單的 bug fix 或小功能，這套方法可能過度設計。但當複雜度跨過某個門檻時，投入時間進行系統化驗證，往往能節省後續數週的調試和返工。
