想像這樣一個場景：一個功能通過了所有單元測試，型別檢查也沒有問題，API 路由存在，UI 頁面也實現了。但當真正的使用者試著使用它時，卻發現什麼都不工作。

這就是「技術上實現」和「使用者能實際使用」之間的巨大鴻溝。

## 為什麼會這樣？

在複雜的 LLM Agent 架構中，一個功能往往涉及多個系統組件的協作——UI 層收集使用者輸入、API 層驗證並處理請求、資料庫層持久化資料、後端服務讀取並處理這些資料，最後才將結果呈現給使用者。只要其中任何一環出現斷裂，整個流程就會失敗。

常見的失敗模式包括：

- **管道斷裂**：UI 成功寫入資料庫，但後續的消費者（例如 PromptComposer）從未讀取過這份資料
- **環境變數缺失**：程式碼引用了某個環境變數，但它在 `.env` 檔案中根本不存在
- **API 驗證差距**：UI 展示了某個選項，卻被 API 的 Zod schema 拒絕
- **無人消費**：功能存在於伺服器，但沒有任何程式碼調用它（沒有 UI、沒有 CLI、沒有外部客戶端）

## 完整流程追蹤法

解決這個問題的方法很簡單：對於提案中的每個功能，都要追蹤完整的**使用者流程**：

```
UI（使用者動作）→ API（HTTP 請求）→ Service（業務邏輯）→ DB（資料持久化）
→ [消費者從 DB 讀取] → Pipeline（處理流程）→ Output（使用者看到結果）
```

逐一檢查每個環節：

1. **UI 是否存在？** 使用者能否觸發這個功能？
2. **API 是否接受請求？** 檢查 Zod schema 的驗證規則是否正確
3. **Service 是否正確寫入資料庫？**
4. **是否有人讀取資料庫中的資料並使用？**（這是最常見的斷裂點）
5. **Pipeline 是否正確處理了資料？**
6. **最終結果是否傳達給使用者？**

除此之外，還要檢查：

- 必要的環境變數是否同時存在於 `.env.shared` 和 Coolify 配置中
- Zod schema 是否接受 UI 可能發送的所有值
- Factory/Provider 是否返回真實實現（而不是樁實現）

## 實際案例

以 Instructions 功能為例，完整稽核清單如下：

- **UI 層**：Settings 頁面存在，CRUD 操作正常 → ✅ 通過
- **API 層**：POST/GET/PATCH/DELETE 路由存在 → ✅ 通過
- **DB 層**：instructions 表存在，Drizzle schema 正確 → ✅ 通過
- **PromptComposer**：有 instructions 參數，預算為 4000 token → ✅ 通過
- **❌ orchestrator-bridge.ts 從未呼叫過 getInstructionsByUser()**

結果就是：`context.instructions` 永遠是 `undefined`，PromptComposer 跳過了指令注入，AI 根本看不到使用者設定的 Instructions。

整個功能在技術上「完整實現」，但對使用者而言是完全不可用的。問題不在任何單一組件，而在於組件之間的連接。

## 何時使用這個方法

- **實現完成後**：多組件功能實現完畢，進行全面驗收前
- **架構審查期間**：提案評審階段，驗證前檢查技術可行性
- **用戶回報問題時**：功能在測試中「正常工作」，但真實使用者反映不可用
- **功能標記為完成前**：確保每個提案中的能力不只是「實現」，而是「使用者可用」

這種追蹤方法特別適合 LLM Agent 這類複雜系統，因為這類系統通常涉及多層次的資料流動和組件交互。投入幾分鐘進行這樣的完整性檢查，可以省去事後尋找「為什麼功能不工作」的數小時除錯時間。
