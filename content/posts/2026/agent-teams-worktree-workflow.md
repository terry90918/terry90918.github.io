在大型開發任務中，如何讓多個 Agent 隊友在同一專案中協作，而不互相干擾？答案是結合 Agent Teams 和 Git Worktree——為每個隊友建立隔離的開發環境，各自進行開發，完成後以 PR 的方式整合回主分支。

## 工作流程概述

這個工作流涉及五個關鍵步驟：

**第一步：Lead 在 develop worktree 接收任務**

Lead（通常是你自己）作為協調者，在主要的 develop worktree 中理解任務需求、規劃工作項目，並準備分配給隊友。這是整個協作的起點。

**第二步：使用隔離 worktree 派遣隊友**

當派遣 Agent 隊友執行工作時，使用 `isolation: "worktree"` 參數建立隔離環境。每個隊友會在自己獨立的 worktree 中開發，確保他們的修改不會影響其他開發者的工作。

**第三步：隊友在隔離環境中獨立開發**

隊友在各自的 worktree 中進行代碼編寫、測試和驗證。由於環境完全隔離，多個隊友可以同時修改不同的檔案或模組，而不用擔心版本衝突或狀態污染。

**第四步：開發完成後發 PR 合併**

隊友完成工作後，從他們的 feature branch 向 develop 分支發起 Pull Request。這個 PR 會包含該隊友的所有修改，並經過 Code Review 流程驗收。

**第五步：develop 驗收通過，PR 到 main 進行正式部署**

當 develop 分支的修改通過測試和 review 後，再向 main 分支發起 PR。合併到 main 會觸發自動部署流程，將新功能推送到正式環境。

## 選擇策略：Agent Teams vs Subagents

不是所有情況都適合用 Agent Teams。以下是決策表：

| 場景                       | 推薦方案     | 原因                 |
| -------------------------- | ------------ | -------------------- |
| 獨立模組、隊友需要互相討論 | Agent Teams  | 需要實時溝通和協調   |
| 快速查詢、結果直接回報     | Subagents    | 單向任務，無需互動   |
| 同檔案修改、存在依賴關係   | 單一 session | 避免衝突，保持一致性 |

簡單來說，當多個隊友的工作相對獨立，但需要在某些決策點溝通時，Agent Teams 是最優選擇。如果只是並行執行完全獨立的查詢或分析，Subagents 更輕量。而當涉及同一檔案的複雜修改時，還是讓單個 Agent 完整處理較為妥當。

## 實踐中的最佳做法

**團隊規模建議**

建議控制在 3 到 5 個隊友，每人承擔 5 到 6 個任務。這個規模既能充分發揮並行優勢，又不會讓協調成本爆炸。

**嚴格的分支管理**

絕對禁止任何人直接 push 到 main 分支。main 連接正式環境，任何直接 push 都會立即觸發生產部署。所有修改都必須先合併到 develop，通過測試和 review 後，再以 PR 的方式進入 main。

**會話持久性**

如果你使用 tmux 來管理 Lead 的工作會話，tmux 可以保持 session 在後台持續運行。這樣即使斷開 SSH 連線，你的開發環境和任務上下文也不會丟失，隊友可以繼續在隔離的 worktree 中工作。

**避免檔案競爭**

協調隊友的工作分配，盡量避免多人同時修改同一檔案。如果確實需要，應該在 Lead 層面預先劃分職責，或者定義清晰的模組邊界，讓每個隊友只關注自己的部分。

## 適用場景

這個工作流特別適合以下情況：

**並行開發多個獨立功能**

當 Terry（或其他用戶）交辦一批需要並行開發的任務時，Agent Teams 可以同時派遣多個隊友分頭進行，大幅加快開發速度。

**跨模組功能開發**

前端、後端、測試可以並行進行。前端隊友開發 UI 元件，後端隊友實現 API 端點，測試隊友準備測試用例，互不阻塞。

**並行 Code Review 和除錯**

當多個 PR 同時等待 review，或多個模組出現問題需要除錯時，派遣多個隊友並行處理，可以快速定位和修復問題。

## 總結

Git Worktree + Agent Teams 的組合，為多代理協作提供了一個清晰、可擴展的框架。通過隔離開發環境和嚴格的分支管理，既保證了代碼品質，又最大化了並行開發效率。關鍵是在適當的場景下選擇這個模式，避免過度工程化，讓協作變得既高效又可靠。
