## 如何發現工具的誤判

在工程工作中，我們經常依賴自動化工具來搜尋檔案、檢查依賴、驗證狀態。但工具有時會給出錯誤的結論——不是因為它們壞了，而是因為在特定場景下（比如 git worktree 路徑）產生了偽陰性（false negative）。最危險的情況是，我們基於這個單一的假陽性結果，做出了一系列串聯的錯誤改動。

### 真實案例：Glob 誤判引發的連鎖反應

我遇過一個典型的例子。Glob 工具回報某個目錄「找不到任何檔案」。基於這個結果，我做出了假設並執行了以下三個步驟：

1. 結論：Skill 目錄不存在
2. 改動：修改 SKILL.md 的參考路徑指向別處
3. 進一步：修改相關的測試以符合新的代碼邏輯

結果呢？三項改動全都錯了。原始的代碼和測試是正確的。誤判本身不是問題，但基於誤判做出的假設，再加上我沒有質疑這個假設，就引發了一連串的錯誤修正。

錯誤的連鎖反應看起來像這樣：

```
工具誤判（假陰性）
    ↓
建立錯誤的假設
    ↓
修改代碼以符合假設
    ↓
修改測試以符合新的代碼
    ↓
虛假的信心：「所有測試都通過了！」
    ↓
錯誤被提交
```

### 驗證第一：在信任工具之前

我學到的教訓很簡單：**當工具說某些東西不存在，而你打算基於此採取行動時，先驗證。**

下面是實踐的具體做法：

**Glob 回報「找不到」？用 ls 驗證：**

```bash
ls /path/to/expected/directory/
```

**還是懷疑？用 Bash 的 stat 命令進一步確認：**

```bash
stat /path/to/file 2>&1
```

這兩個命令會給你確定的答案。如果 Glob 說找不到，但 ls 和 stat 都顯示檔案存在，那麼問題很可能出在工具的特定場景支援（比如 git worktree 路徑的相對路徑解析）。

### 測試失敗時的決策樹

當你修改代碼後測試失敗時，很容易衝動地修改測試本身以使其通過。但這就是錯誤連鎖的開始。你需要回到源頭問自己：

```
我的修改後測試失敗
    ├─ 測試在驗證正確的業務邏輯嗎？
    │   ├─ 是 → 我的代碼改動有問題，應該修復代碼
    │   └─ 不確定 → 驗證促使我做出這個改動的前提假設
    │       ├─ 前提是對的 → 測試可能需要更新
    │       └─ 前提是錯的 → 回滾所有改動
    └─ 測試已過時嗎（業務需求改變了）？
        ├─ 是 → 更新測試（並記錄為什麼改變）
        └─ 否 → 我的代碼有問題，修復代碼
```

關鍵是：**如果你懷疑的是導致改動的前提（而非改動本身），那就回到源頭驗證前提，而不是繼續修改下去。**

### Glob 工具的已知限制

根據實踐經驗，以下情況容易出現問題：

- 在 git worktree 目錄中使用相對路徑進行 Glob 查詢時，可能返回偽陰性
- 改用絕對路徑通常能解決問題
- 當不確定時，總是用 ls 或 stat 進行交叉驗證

### 何時應該套用這個診斷模式

這個模式在以下情況特別有用：

1. **搜尋工具報告「找不到」**，而你準備基於此刪除參考、跳過實現，或修改測試
2. **修改代碼後測試失敗**，你正在考慮修改測試本身來「修正」

### 核心原則

不要讓單一工具的結果驅動你的決定。特別是當這個結果要求你修改測試或刪除「看起來無用」的代碼時。工具很聰明，但它們也有盲點。在採取行動前，用另一個角度驗證前提——無論是用命令列工具、代碼檢查，還是簡單的邏輯思考。

因為工具誤判本身只會浪費幾分鐘。但基於誤判連鎖發生的錯誤修正，卻可能在代碼庫中潛伏很久。
