每當團隊啟動大型重構時，總會面臨一個共同的危機：把本質完全不同的工作混在一起執行。刪除冗餘的代碼、改名統一欄位、保留因業務需求不得不留下的邏輯——這三件事看似相關，實際上需要不同的風險評估、完全不同的測試策略，甚至截然不同的回滾方案。結果就是，工時估算翻倍、測試破口四處出現、整個提案失焦。

我稱之為「三分法」：在重構執行之前，先把所有待改項分成三類——**刪除**、**改名**、**保留**。聽起來簡單，但實踐時需要問自己三個簡單但關鍵的問題。

## 三個判斷問題

當你列出所有待改項時，對每一項問：

**第一個問題：這東西被新架構完全取代了嗎？**

如果答案是「是」，這是**純刪**——翻譯層、舊的 feature flag、已死的代碼。它的存在零價值，移除就獲得收益。風險低、目標明確。如果答案是「否」，繼續問。

**第二個問題：它的值跟新架構同源嗎？**

換個說法：新舊兩邊指向同一份資料，只是名稱不同？比如舊系統叫 `callId`，新系統叫 `tool_use_id`，但都指向同一個東西。這是**改名**。沒有功能收益，只是命名統一。這一類需要單獨評估——值得投入嗎？

**第三個問題：外部系統有對等概念嗎？**

如果 SDK 或上游服務沒有這個欄位，而它解決的是業務層面的問題，那就是**保留**。這是 anti-corruption layer 的價值所在。它不冗餘、不是命名差異，就是業務邏輯。

## 為什麼要分開執行？

**刪除類**應該首先執行，單一 commit，風險最低。沒有功能損失，只有代碼行數減少。

**改名類**應該獨立提案。既然沒有功能收益，要決定值不值得——是否命名一致性的潔癖足以正當化這筆工時？

**保留類**在 design 文件裡明確標註「為什麼保留」。列出對應的 SDK 欄位，或者說明沒有等價概念。這樣下一次重構時就不會再被誤認為冗餘。

三類分開，每類用自己的工時估算、測試範圍、風險評級。結果就是提案不再失焦，利益相關者能精確衡量每一步的代價。

## JurisLM 的真實案例

我們本來計畫一口氣刪掉 `EventBridge` 類、遷移所有的 `callId` 為 `tool_use_id`。乍一看是「統一重構」。

用三分法重新分類：

**純刪的項目：**

- `EventBridge` 類（250 多行）—— SDK passthrough 已經取代它，零價值
- `SseEvent` union —— 自訂協議，SDK 原生事件已代替
- `mcp_call_start/end/result` 三元組 —— 自訂拆分，SDK 本身就是 `agent.mcp_tool_use` + `agent.mcp_tool_result`
- `MANAGED_AGENT_SDK_PASSTHROUGH` 環境變數 —— 遷移時的鷹架，任務完成後無用

**改名的項目：**

- `McpCallRecord.callId` —— 與 SDK 的 `tool_use_id` 同源（`sdk-event-handler` 已翻譯）
- `McpCallRecord.toolName` —— 與 SDK `name` 同源
- `McpCallRecord.results` —— 與 SDK `parsed_results` 同源

**保留的項目：**

- `RecoveredMcpCall.intro` —— 從前一個 `agent.message` 跨事件擷取的語意，SDK 沒有
- `RecoveredMcpCall.input` —— UI fallback 欄位，shape 是業務決定
- `RecoveredMcpCall.status: "interrupted"` —— 缺失事件推論，SDK 不理解這個概念

**決策**：Phase I 只做刪除類（Option X），改名類延後另立議題。

**結果**：工時估算從 8-10 小時縮減到 5 小時；風險從「中」降到「低」；實際執行只需 4 個 commit，移除 1692 行、新增 251 行，三個 CI/CD 檢查全綠。如果一開始混在一起，我們現在還在調試 React key 和測試斷言。

## 何時使用這個方法

看到這些信號時，立刻啟動三分法：

- **「刪除 X 類 **並且** 改名 Y 欄位」** —— 兩件事，要拆
- **「Phase N 收斂 dual-shape」** —— "dual-shape" 常同時指協議和命名兩件事，容易混淆
- **「大改 / 重構」提案的 "What Changes" 列出 5 項以上** —— 很可能混了三類
- **工時估算突然膨脹、測試破口比預期多** —— 這是混類的信號
- **用戶 push back「為什麼要刪 / 為什麼不從 SDK 取」** —— 說明不同類的風險被當成同一個處理了

## 關鍵收穫

大型重構不是一件事，是三件事，它們恰好都涉及改變代碼。但改變的本質完全不同：

- **刪除** = 移除無價值物
- **改名** = 調整命名風格（無功能變化）
- **保留** = 承認邊界，記錄決策

分開執行，每一步都變得可控、可測、可回滾。下一次有人建議再改時，你已經有案可查了。
