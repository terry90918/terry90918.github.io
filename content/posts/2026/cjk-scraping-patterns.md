在處理中文（CJK 字符）的網頁爬蟲和文字比對時，許多看似簡單的問題其實隱藏著細微的陷阱。我在開發與政府網站互動的爬蟲工具時，遇到了兩個核心挑戰，今天想分享解決這些問題的實戰經驗。

## 中文子字串匹配的詞長閾值問題

當你需要進行同義詞擴展或關鍵字搜尋時，最直接的做法就是用 `String.includes(term)` 來檢查目標字串中是否包含某個詞。但這種方式有個致命的缺點：**它沒有詞邊界的概念**。

舉個例子。假設你想搜尋「退貨」相關的法律條款，於是寫了這樣的邏輯：

```typescript
const keywords = ['退', '貨', '退貨']
const text = '本合約金額由雙方同意...'

keywords.forEach((keyword) => {
  if (text.includes(keyword)) {
    console.log(`找到 ${keyword}`)
  }
})
```

看似合理，但當關鍵字是單個中文字「金」時，它會意外地匹配到「合約**金**額」中的「金」。這種誤匹配在大量文件中會產生大量噪聲，導致搜尋精度大幅下降。

另一方面，如果你過度保守，直接排除所有單字元詞，又會漏掉許多有意義的 2 字元中文詞彙，比如「車禍」、「噪音」、「押金」等等。

**解決方案很簡單：設定詞長閾值為 2 字元**。

```typescript
const keywords = ['退', '貨', '退貨', '押金']
const minTermLength = 2

keywords.forEach((keyword) => {
  if (keyword.length >= minTermLength && text.includes(keyword)) {
    console.log(`找到 ${keyword}`)
  }
})
```

為什麼是 2 字元？因為中文的法律術語、行業用語絕大多數都是 2-4 字元的詞彙組合。而單字元的匹配幾乎都是無意義的重疊。通過這個簡單的篩選，你既保留了有實用價值的短詞，又剔除了大部分無謂的誤匹配。

## 政府網站爬蟲的「假設陷阱」

如果說上一個問題是文字處理層面的挑戰，那麼政府網站的爬蟲開發則充滿了對網站結構的錯誤假設。

我在爬取台灣司法院、法務部等政府機關的公開資料時，經常會做出這樣的假設：

**假設 1：URL 格式遵循某種規律**  
比如，看到一個分頁 URL 是 `lp-1361-1-1-500.html`，我會假設每頁返回 500 筆記錄，下一頁是 `lp-1361-1-1-1000.html`。但實際上，URL 參數並不總是這樣起作用——有時候伺服器會忽視你的分頁參數，無論怎麼改 URL，都返回同樣的內容，甚至是完全無關的頁面。

**假設 2：頁面爬取層次是可預測的**  
我會假設需要「列表頁 → 詳細頁 → 下載連結」三層爬取。但很多時候，列表頁本身就直接包含了所有下載連結，詳細頁根本不需要訪問。這種假設會導致大量無用的 HTTP 請求，浪費時間和頻寬。

所以，**如何避免這些假設失敗？答案是 curl-first pattern**。

## curl-first Pattern：先驗證，再自動化

不要一開始就寫爬蟲。相反，先手動探索網站結構：

**步驟 1：下載真實頁面樣本**

```bash
curl -o page1.html "https://gov-site.example.com/list?page=1"
curl -o page2.html "https://gov-site.example.com/list?page=2"
curl -o page3.html "https://gov-site.example.com/list?page=3"
```

下載 2-3 個真實頁面，這些是你開發爬蟲的真實依據。

**步驟 2：用 grep 觀察 DOM 結構**

```bash
grep -o 'data-title="[^"]*"' page1.html
grep -o 'class="[^"]*link[^"]*"' page1.html
grep -o 'href="[^"]*\.pdf"' page1.html
```

通過正則搜尋，快速定位你關心的 HTML 元素（如 `data-title` 屬性、特定 class、下載連結）。這一步會讓 DOM 結構一目了然，遠比盲目猜測有效。

**步驟 3：確認分頁機制**

觀察多個頁面，確認分頁如何工作：

- 是 URL 參數（`?page=1`, `?page=2`）？
- 是 AJAX 請求（用開發者工具 Network 標籤確認）？
- 是否有固定的頁數限制（比如只有 5 頁）？

**步驟 4：才開始寫爬蟲**

現在你對網站結構有了充分理解，就可以寫出可靠的爬蟲邏輯，而不是基於錯誤的假設。

## 實踐應用

這兩個模式特別適合：

- **中文/CJK 文字的關鍵字匹配**：同義詞擴展、全文搜尋、query 處理
- **爬取台灣政府網站**：司法院判決書系統、法務部公告、立法院提案等
- **任何需要子字串匹配 CJK 文字的場景**：文件分類、內容過濾、資料清理

關鍵是要**先驗證假設，再寫程式**。對於中文文字處理，別被單字元匹配騙了；對於網站爬蟲，別盲目猜測 URL 和 HTML 結構。curl 和 grep 這兩個簡單的工具，往往比幾小時的開發時間更有價值。
